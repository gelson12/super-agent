# Bridge Office вЂ” Living Bot Simulation

A 2D top-down simulation of the Bridge AI bot ecosystem rendered as a 3-floor office. Each Bridge bot (CEO, Chief of Staff, Researcher, CSO, Cleaner, PM, Marketing, Finance, Website, Crypto + ambient agents) appears as an animated sprite that walks the floors, gathers for meetings, and idles in lounges/coffee/ping-pong/snooker zones.

Live at: `https://super-agent-production.up.railway.app/static/office_sim/index.html` (after the next deploy).

## What it does

- **3 floors** with proper room layouts (entry/operations on L1, executive/focus on L2, social/relax on L3) вЂ” backed by hand-mapped 64Г—40 tile grids.
- **A\*** pathfinding per floor, plus inter-floor routing via stairs (no teleporting between levels).
- **Collision** against walls, desks, room boundaries; rooms enter via explicit door tiles.
- **Sprite animation** by alternating stand+walk frames per direction (8-direction sprites: stand-L/R/U/D + walk-L/R/U/D, sliced from 192Г—128 cells in five 1536Г—1024 multi-character sheets).
- **Daily cadence** matching the real Bridge bot cron schedule: 07:30 Researcher morning intel, 08:00 CoS day-open briefing in the L2 conference room, 08:30 CSO risk review, 12:30 midday blocker, 14:00 informal coffee chat, 16:00 CEO exec summary in L3 conference, 17:30 ping-pong break, 20:00 day-close, 22:00 Cleaner nightly hygiene. Sim runs at **60Г— real time** so a "day" compresses to 24 minutes.
- **Meeting reservations** вЂ” picked rooms are locked while a meeting runs; a 5-minute pre-gather walks participants to anchor seats; meeting ends в†’ bots disperse в†’ reservation cleared.
- **Idle social behaviour** вЂ” bots not in a meeting roll a per-bot affinity list every 30 s (60% stay at desk, 40% wander).
- **Live data overlay** (toggle: LIVE | DEMO) вЂ” polls `/n8n/workflows` every 30 s (active workflows light their owning bot's halo), `/dashboard/agents/status` every 5 s (connectivity check), subscribes to `/activity/stream` SSE (events tail into the HUD log; matched bot's task-label updates).
- **HUD** вЂ” bot list with state pip + task label, scrolling activity log, upcoming-meetings list, status bar with mode/conn/fps/bot-count, click any bot row or canvas tile to focus.

## Editing the world

### Move a bot's desk or change a sprite

Open `data/bots.json`. Each bot has:
- `sheet` (`sheet_1`..`sheet_5`) вЂ” which sprite sheet to use
- `row` (0..7) вЂ” which character row within that sheet
- `desk.floor` and `desk.tile` вЂ” must be a real anchor in the matching floor JSON
- `affinity` вЂ” list of zone types the bot prefers when idle

If you change a bot's sprite to the wrong character, edit `row` (each sheet has 8 character rows; check the sprite sheet PNG visually to find the right row for the look you want).

### Reshape a floor

Edit `build_floors.py` (compact high-level layout) and run:
```
python build_floors.py
```
That regenerates `data/floor*.json`. The world loader builds the actual tile grid by stamping `obstacles` rectangles over a default-walkable canvas, then auto-walling the perimeter.

### Add a meeting

Append to `data/schedule.json` under `daily`. Required fields:
```json
{
  "id": "unique-id",
  "time": "09:30",
  "label": "Human-readable label",
  "participants": ["ceo", "chief_of_staff"],
  "preferredRoomType": "conference_room",
  "durationMs": 480000
}
```
Allowed `preferredRoomType`: `meeting_room | conference_room | lounge | coffee | focus | tv_lounge | ping_pong | snooker | balcony`.

## Asset pre-processing

Before deploying, run:

```
python preprocess_assets.py
```

This produces, for each of the 5 sheets and 3 floors:

1. `assets/sprites/sheet_N_alpha.png` вЂ” the source sheet with **near-white pixels alpha-keyed to transparent** (R, G, B all в‰Ґ 235 в†’ О±=0; with a 6-pixel feather for clean silhouette edges). The runtime prefers `*_alpha.png` and falls back to the original `*.png` if the preprocessed file isn't there.
2. `data/floorN_overlay.json` вЂ” a list of **PNG-derived blocked tiles** (cells that visibly contain dark furniture/walls). The script samples each 64Г—40 grid cell against the floor PNG: a tile is added to the overlay only if mean brightness < 95 AND в‰Ґ 62% of its pixels are dark. A connectivity check rejects any candidate that would sever a required anchor from the spawn вЂ” so the overlay can't break navigation. Typical run: ~700вЂ“1000 extra blocks accepted per floor, 20вЂ“60 rejected. The runtime applies the overlay on top of the hand-mapped obstacles.

Re-run `preprocess_assets.py` whenever you swap a floor PNG or sprite sheet.

## Walk-cycle animation вЂ” current state and upgrade path

**v1 (shipping now):** each bot has 8 directional cells (stand-L/R/U/D + walk-L/R/U/D). The renderer alternates between the stand and walk frame for the bot's current facing every 220 ms, plus a procedural vertical sin-bob at 4 Hz with a 3-px amplitude. With single-frame walk poses, that bob is what reads as "stepping". Drop shadow stays planted on the ground while the body lifts.

**v2 вЂ” per-bot horizontal strips with mirrored leg cycle (current):**
Each bot in `bots.json` can declare a `botSprite: "<file>.png"` field. The runtime loads `assets/sprites/bots/<file>.png` (preferring the `_alpha.png` produced by `preprocess_assets.py`) and treats it as a **single-row 8-cell strip** in the standard column order (stand-L/R/U/D + walk-L/R/U/D). Aspect ratio is preserved вЂ” strips can be 177Г—177, 177Г—125, etc.

The renderer cycles a **4-phase walk** at ~180 ms per phase:
- phase 0 в†’ stand
- phase 1 в†’ walk (foot forward)
- phase 2 в†’ stand
- phase 3 в†’ walk **horizontally mirrored** (opposite foot forward) вЂ” for left/right only

Mirroring synthesises an "opposite-leg" pose from a single walk frame, giving a natural alternating-stride look without extra art. Up/down keep the 2-phase stand/walk + procedural bob.

Bots without a `botSprite` field fall back to the multi-bot grid (`sheet_1..5`).

**v3 вЂ” individual per-frame PNGs (BEST вЂ” already wired up):**
The cleanest path. Drop one PNG per (direction, walk frame) into a per-bot folder, with real per-pixel alpha (no white-key heuristic needed). The runtime auto-detects them and skips the strip/grid fallbacks.

The walk cycle per direction becomes `stand в†’ walk_1 в†’ walk_2 в†’ вЂ¦ в†’ walk_N в†’ stand в†’ вЂ¦`, period = N+1 phases Г— 180 ms. With N=4 frames per direction you get a true alternating-leg stride in **all 4 directions**, no mirror tricks. Up/down strides finally look real.

For each bot, `<bot_id>` matches the `id` field in [`data/bots.json`](data/bots.json) вЂ” `ceo`, `cto`, `coo`, `crypto`, `marketing`, `chief_of_staff`, `cso`, `researcher`, `cleaner`, `pm`, `finance`, `website`, `scholar`, `nova`, `writer`. Drop in only the bots/directions you have and the loader silently skips the rest, falling back to the strip or grid for missing pieces.

**How to cut from your existing per-bot reference renders:**
Each `animated_office_bots/Bridge_<bot>_bot/` folder contains 12-15 alternative pose renders (`001_*.png`, `002_*.png`, вЂ¦). Pick:
- one stable "facing X" pose for each of the 4 stand frames
- 2-4 mid-stride poses per direction for the walk frames (foot-forward, mid-step, foot-back, mid-step)
- crop to the character's bounding box, save with transparent alpha, name per the layout above

## Known compromises after v2.1

- **Sprite-sheet row mapping is a guess** when a bot falls back to the multi-bot grid вЂ” edit `row` in `bots.json` until it visually matches.
- **Hand-mapped tile maps + PNG-derived overlay** combined still won't be pixel-perfect against every furniture corner. Press `g` to overlay the navigation graph and see exactly which tiles the engine considers walkable; tweak `build_floors.py` or `FURNITURE_MEAN_MAX/FURNITURE_DARK_FRAC` in `preprocess_assets.py` as needed.
- **Sprite scale 64 px** at 1280-canvas baseline вЂ” readable but proportional to the office. Tune `SPRITE_DRAW_W/H` in `renderer.js`.
- **Demo cadence not weekend-aware** вЂ” Mon/Tue weekly events don't fire if the wall clock isn't on those days.
